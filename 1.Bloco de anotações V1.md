# Funcionalidade: Tela de Anotações – V1

## 0. Contexto

O aplicativo já possui módulos de Painel, Agenda, Processos, Histórico e Configurações, com navegação pelo menu lateral.[page:1]  
Já existe uma rota `/anotacoes` com um bloco de notas funcional básico (rich text com negrito, itálico, sublinhado, cores e alinhamentos), mas:

- Faltam controles de **tamanho de fonte** e **marcadores de parágrafo** (bullets, setas, checklists).  
- A lista de notas não organiza visualmente as notas mais recentes e antigas como desejado.  
- A UX geral de leitura/edição pode ser melhorada para uso diário (cards, destaque das últimas notas etc.).[page:1]  

Este PRD define a primeira versão “completa” da tela de Anotações.

---

## 1. Objetivo

Prover uma tela de **Anotações** onde o usuário:

- Cria e edita notas com um **editor rico** voltado a organização de texto (listas, checklists, formatação básica).[page:1]  
- Visualiza suas notas de forma organizada, com destaque para as **5 mais recentes** e as demais recolhidas/compactadas.  
- Mantém as funcionalidades existentes (busca, tags, exportação, compartilhamento, ditado, autosave) sem regressões.[page:1]  

---

## 2. Escopo

### 2.1. Dentro do escopo

- Melhorias na **rota `/anotacoes`**:  
  - Toolbar do editor de texto.  
  - Organização visual da lista de notas (recentes vs antigas).  
  - Ajustes de UX (indicador de nota selecionada, responsividade, layout em cards).  

### 2.2. Fora do escopo

- Qualquer alteração nas rotas `/principal`, `/agenda`, `/processos`, `/historico`, `/configuracoes`.  
- Mudanças de modelo de dados além do estritamente necessário para suportar novos tipos de lista/checklist (se já existir suporte, reutilizar).  

---

## 3. Editor de Anotações – Toolbar

### 3.1. Situação atual

O editor já oferece:

- Negrito, itálico, sublinhado.  
- Alinhamentos (esquerda, centro, direita, justificado).  
- Cor do texto (ou similar).  

### 3.2. Novos controles obrigatórios

Adicionar à toolbar:

1. **Tamanho de fonte**  
   - Controle de tamanho com pelo menos 3 níveis:  
     - Pequeno  
     - Padrão  
     - Grande  
   - Pode ser implementado como dropdown (P, Normal, G) ou ícones com “A- / A / A+”.  
   - Aplicação deve ser compatível com o editor já usado (por exemplo, usando classes ou tags de heading quando fizer sentido).

2. **Lista com bullets (não ordenada)**  
   - Botão para criar lista não ordenada (ul/li):  
     - Cada Enter cria um novo item.  
     - Voltar no início de item vazio encerra a lista (comportamento padrão de editores rich text).  

3. **Lista com marcadores tipo seta**  
   - Botão que cria lista com prefixo visual de seta (ex.: “→”).  
   - Pode ser implementado como um estilo alternativo de lista (ex.: usando ícone ou pseudo-elemento CSS).  
   - Cada item deve iniciar com uma seta visual clara, sem quebrar o layout existente.

4. **Checklist com caixas de verificação**  
   - Botão que cria uma **lista de checklist** com checkboxes:  
     - Cada linha é um item com uma checkbox clicável.  
     - Estado marcado/não marcado deve ser persistido junto com o conteúdo da nota.  
   - Se o modelo atual de notas já suporta algum tipo de estrutura de checklist, reutilizar; senão, implementar de forma consistente com o editor em uso.

### 3.3. Critérios de aceite (editor)

- Usuário consegue:  
  - Ajustar o tamanho da fonte em trechos selecionados.  
  - Criar listas com bullets.  
  - Criar listas com setas.  
  - Criar checklists com caixas marcáveis.  
- Após salvar e recarregar a nota, as formatações permanecem corretas.  

---

## 4. Organização das notas – Recentes vs antigas

### 4.1. Requisito principal

- As **5 notas mais recentes** devem ser exibidas de forma **expandida e em destaque**.  
- As notas mais antigas devem aparecer **recolhidas/compactadas** numa lista, podendo ser expandidas sob demanda.[page:1]  

### 4.2. Layout sugerido

Seção 1: **Notas recentes**

- Título da seção: “Notas recentes” (ou similar).  
- Exibir 5 notas mais recentes (por data de atualização).  
- Cada nota como **card** com:  
  - Título.  
  - Data de atualização.  
  - Tags (se existirem, como chips/pílulas).  
  - Um trecho inicial do conteúdo (preview).  
- Card clicável abre a nota no editor (ou marca como selecionada, mantendo o editor ao lado/dentro da mesma tela, conforme já implementado).  

Seção 2: **Outras notas**

- Título da seção: “Outras notas” (ou “Notas anteriores”).  
- Lista compacta (por exemplo, linha por nota) com:  
  - Título.  
  - Data.  
  - Ícones de tag ou estado (se houver).  
- Cada linha expansível: clique abre a nota no editor e, opcionalmente, expande o card.  

### 4.3. Comportamento esperado

- A separação “5 mais recentes” x “outras” deve ser recalculada automaticamente conforme novas notas são criadas ou existentes são editadas (base em `updated_at`).  
- Ao abrir a tela `/anotacoes`, uma das notas recentes deve estar pré-selecionada (por exemplo, a mais recente).  

### 4.4. Critérios de aceite (lista)

- As 5 notas mais recentes aparecem claramente separadas e em destaque.  
- Notas mais antigas ficam recolhidas, mas acessíveis com clique.  
- Navegar entre notas não quebra o editor nem o layout.  

---

## 5. UX e estética da tela

### 5.1. Layout em cards e responsividade

- Aplicar **layout em cards** para as notas recentes:  
  - Bordas suaves, sombra leve, uso consistente das cores já adotadas no app (para manter identidade visual).  
- Garantir que a tela de Anotações seja **responsiva**:  
  - Em telas menores, os cards podem virar lista vertical.  
  - O editor não deve ficar espremido nem extremamente largo (limitar largura máxima razoável).

### 5.2. Indicador de nota selecionada

- Destacar visualmente a nota atualmente em edição:  
  - Card com borda acentuada ou cor de fundo levemente diferente.  
  - Na lista de “Outras notas”, linha selecionada também deve ter destaque.  

### 5.3. Tags e chips

- Se o modelo de notas já possui **tags**, padronizar sua visualização:  
  - Exibir tags como chips/pílulas coloridas nos cards.  
  - Manter o mesmo padrão de cores/tipografia que o app já usa em outras áreas (se houver).  

### 5.4. Manter recursos existentes

- Manter sem regressão:  
  - Busca por título/texto/tag/data.  
  - Exportação (TXT/MD).  
  - Compartilhamento via WhatsApp.  
  - Ditado por voz (se já implementado).  
  - Salvamento automático.  

---

## 6. Navegação e integração

- O item “Anotações” no menu lateral continua levando à rota `/anotacoes` (já existente).[page:1]  
- Nenhuma mudança é necessária na navegação global, apenas dentro da própria tela de Anotações.  

---

## 7. Critérios de aceite gerais

Considerar a implementação da tela de Anotações V1 concluída quando:

1. **Editor**  
   - Toolbar contém tamanho de fonte, bullets, lista com setas e checklist, todos funcionais.  
   - Estado de checklists e formatações é persistido e restaurado corretamente.  

2. **Lista e layout**  
   - As 5 notas mais recentes aparecem em destaque, expandidas.  
   - Notas antigas estão em seção separada, recolhidas/compactadas.  
   - Nota selecionada tem indicação visual clara.  
   - Tela é utilizável em desktop e em telas menores (layout adaptável).  

3. **Funcionalidades existentes**  
   - Busca, tags, exportação, compartilhamento e autosave continuam funcionando.  

4. **Restrições**  
   - Nenhuma rota além de `/anotacoes` foi alterada.
